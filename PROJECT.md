# TableMind — AI-официант для ресторанов

## What It Is

PWA-приложение, которое гость открывает по QR-коду на столе в ресторане. Три режима работы: просмотр меню, меню с AI-официантом (рассказывает о блюдах прямо в карточках + чат), и интерактивный подбор комплекса через мини-тест. Заказ передаётся официанту через Telegram. Мульти-агентная архитектура: каждый AI-агент отвечает за свою задачу, верификаторы асинхронно проверяют точность ответов.

## The Problem

- Официант перегружен в часы пик — гость ждёт 5-10 минут просто чтобы заказать
- Официант не помнит все составы и аллергены — юридический и репутационный риск
- Гость стесняется задавать вопросы → заказывает «что попроще» → средний чек ниже
- Иностранные/туристические гости не понимают меню → уходят
- Существующие решения (планшеты Британика) — это просто цифровое меню без интеллекта. Гость всё равно зовёт официанта, когда хочет что-то узнать или подобрать
- Ресторатор не знает, что гости спрашивают → не может улучшать меню

## Target Users

**Гость ресторана:**
- Турист (не знает кухню, не читает по-русски) → нужен мультиязычный AI
- Технически любопытный человек → хочет интерактив, новые технологии
- Человек, который не может дождаться официанта → хочет изучить меню и выбрать сам
- Человек с диетическими ограничениями → нужна точная информация о составе

**Ресторатор (покупатель):**
- Ресторан/кафе среднего сегмента в Калининграде (средний чек 1 000–3 000 ₽)
- 15-50 столов, 30-150 позиций в меню
- Проблема: нехватка официантов, сезонные пики, иностранные гости
- Не готов к полному самообслуживанию (не фастфуд), но хочет разгрузить персонал

**НЕ наш клиент:**
- Фастфуд / кофейни у стойки (заказ за 30 секунд, AI не нужен)
- Люкс с личным сомелье (гость платит за человека)
- Сети 5+ точек с юротделом (долгий цикл)

## Success Criteria (v1)

- Гость сканирует QR → PWA открывается за <2 секунды на мобильном
- Гость может просмотреть всё меню с фильтрами (категории, теги)
- Гость нажимает на блюдо → AI рассказывает о нём прямо в карточке (инлайн, не уходя в чат)
- Гость открывает чат → AI рекомендует блюда из реального меню, не выдумывает
- Гость собирает «мой выбор» → жмёт «позвать официанта» → Telegram-уведомление с номером стола и списком
- Верификатор асинхронно проверяет каждый ответ AI на точность (аллергены, состав, цены)
- Гость может общаться с AI голосом на своём языке (Web Speech API)
- Демо работает на тестовом ресторане и готово для рассылки потенциальным клиентам

## Три режима приложения

### Режим 1: Просто меню
Каталог блюд с фото, ценами, описанием, тегами (веган/острое/без глютена). Фильтры по категориям и диетам. Без AI.

### Режим 2: Меню с AI-официантом
Всё из режима 1 + интерактивные элементы:
- Иконка на карточке блюда → нажал → AI рассказывает о блюде прямо там (инлайн, без перехода в чат)
- Полноценный чат для сложных вопросов («что к стейку?», «есть ли в пасте орехи?», «что-нибудь лёгкое до 600 ₽»)
- **Голосовой режим:** гость говорит с AI-официантом голосом (Web Speech API). AI понимает и отвечает на языке гостя — русский, английский, немецкий и др. Мультиязычность из коробки, без ручной настройки

### Режим 3: Интерактивный подбор
AI ведёт мини-тест: спрашивает о предпочтениях, что уже ели, любимые вкусы. По результатам собирает готовый комплекс (закуска + основное + напиток + десерт). Несколько вариантов на выбор — иллюзия выбора, не навязывание.

## Мульти-агентная архитектура AI

| Агент | Специализация | Когда работает |
|---|---|---|
| Официант | Рекомендации, рассказ о блюдах, подбор по запросу | Режим 2 (инлайн + чат) и режим 3 |
| Тестировщик | Ведёт мини-тест, формирует комплекс по результатам | Режим 3 |
| Менеджер | Техподдержка, вопросы о заведении (время работы, бронь, Wi-Fi) | Всегда доступен |
| Верификатор | Проверяет ответы официанта на точность (аллергены, состав, цены) | Асинхронно, после каждого ответа |

Принцип: каждый агент делает одно дело хорошо. Добавляем функциональность через новых агентов, не раздувая промпты существующих.

## Бонусные фичи

### Голосовой AI-официант
Web Speech API (встроен в браузер, бесплатно). Гость говорит на своём языке → AI понимает и отвечает. Мультиязычность без ручной настройки — браузер определяет язык речи автоматически. Особенно ценно для туристического Калининграда.

### «Я тут уже был» — память гостя
localStorage / IndexedDB запоминает предпочтения анонимно (без регистрации). При повторном визите: «В прошлый раз вам понравилось острое — попробуйте новый том-ям». Дополнительно запрашиваем `navigator.storage.persist()` для надёжности. Если данные всё же очистятся (iOS Safari может через 7 дней) — ничего критичного не ломается, просто не будет персонализации. Приятный бонус, не критичная фича.

## Hard Constraints

- **Команда:** 1 человек (разработчик)
- **Срок MVP:** ~1 неделя до рабочего демо + начала рассылки клиентам
- **Бюджет:** небольшой, есть (не нулевой)
- **AI должен работать в РФ:** OpenRouter как прокси к Claude Haiku
- **Точность AI:** жёсткая привязка к данным меню. AI НЕ может выдумывать блюда, цены, состав. Только то, что в каталоге
- **Скорость загрузки:** критична — гость открывает с QR на телефоне, ждать не будет
- **Меню обновляется редко:** раз в 2-3 месяца, допустимо обновлять JSON вручную

## Стек (на основе ZalAssist)

| Компонент | Решение | Почему |
|---|---|---|
| Фронтенд (PWA) | SvelteKit + Tailwind + DaisyUI | Проверено на ZalAssist, быстрый, лёгкий |
| Поиск | MiniSearch (в браузере) | Офлайн, мгновенный, с синонимами |
| Каталог меню | Статический JSON | Обновляется редко, не нужна БД |
| AI | OpenRouter → Claude Haiku | Работает в РФ, дёшево (~0.001-0.003$ за диалог) |
| API (AI) | Vercel Edge Functions | Стриминг ответов |
| Фронт-хостинг | GitHub Pages | Бесплатно |
| Уведомления | Telegram Bot API | Вызов официанта с номером стола и списком |
| Дизайн | Новый, ресторанный | ZalAssist — только архитектурный паттерн |

## Архитектура

```
[Гость] → QR на столе → PWA (GitHub Pages)
               ↓
         [SvelteKit PWA]
         ├→ catalog.json (меню, загружается при старте)
         ├→ MiniSearch (поиск/фильтры в браузере)
         ├→ Режим 1: статичный просмотр
         ├→ Режим 2: инлайн AI + чат
         │   ├→ selectItemsForAI() → ~50 релевантных блюд
         │   ├→ Vercel Edge → OpenRouter → Claude Haiku
         │   ├→ Стриминг ответа
         │   ├→ Парсинг ID блюд из ответа → карточки
         │   └→ Верификатор (асинхронно)
         ├→ Режим 3: тест → комплекс
         │   └→ Тестировщик-агент ведёт сценарий
         └→ «Позвать официанта» → Telegram Bot API
              → Сообщение: стол №, список выбранного
```

## Known Unknowns

- Оптимальный формат мини-теста (сколько вопросов, какие)
- Дизайн: какой визуальный стиль для ресторанного сегмента
- Как именно верификатор помечает неточные ответы (молча правит? показывает предупреждение?)
- Какие дополнительные AI-агенты понадобятся помимо основных четырёх
- Формат демо для рассылки (ссылка на тестовый ресторан? видео?)

- Кастомный дизайн под каждый бренд

**Фреймворк:** 9 900 ₽/мес — это *honesty fee* за то, чтобы продукт жил и работал. Не агентство, не менеджер аккаунта, не круглосуточная поддержка. Я solo-разработчик, говорю об этом прямо.

**Математика:**
- Revenue per client: `9 900 × 12 = 118 800 ₽/год`
- 10 активных клиентов ≈ 1.19 млн ₽/год валовой; ~1.12 млн ₽/год чистыми после OpenRouter (~5 000 ₽/мес) и хостинга (~1 000 ₽/мес)
- Точка безубыточности (100 000 ₽/мес чистыми) ≈ 11 активных клиентов = минимум 14 подписаний с учётом 3 бесплатных пилотов

**Риски модели:**
- Лимит самозанятого 2.4 млн ₽/год = потолок ~20 клиентов → дальше ИП на УСН или ООО
- Churn: каждый ушедший клиент режет MRR сразу (vs разовая продажа где деньги уже получены)
- OpenRouter теперь за мой счёт → надо жёстко закладывать лимиты в промптах и rate-limit на фронте
- Обновления меню × клиенты = скрытая нагрузка; нужна автоматизация (n8n парсинг) к порогу в 10+ клиентов
