# Интеграция с POS-системами

## Принцип: только READ, никогда WRITE

Мы **только читаем** меню из POS. Никогда не отправляем заказы, не трогаем кассу, не пишем в базу POS. Это наш главный принцип «сделал-отдал-забыл»: нет write = нет ответственности за сбои.

---

## Уровни интеграции

### Уровень 0. Google Sheets (Basic-пакет)
- Клиент сам ведёт меню в таблице
- Колонки: категория, название, описание, цена, фото URL, аллергены, теги
- Cloudflare Worker читает Sheets API раз в сутки → кеширует в KV
- **Сложность:** нулевая
- **Что нужно от клиента:** заполненная таблица

### Уровень 1. Poster POS (Pro-пакет)
- **API:** REST, открытая документация — https://dev.joinposter.com/
- **Авторизация:** OAuth2, токен из личного кабинета
- **Эндпоинты:**
  - `GET /menu.getProducts` — все блюда с ценами, фото, категориями
  - `GET /menu.getCategories` — категории меню
  - `GET /storage.getIngredients` — состав (для аллергенов)
- **Сложность:** низкая, самый простой POS для интеграции
- **Что нужно от клиента:** токен из Poster кабинета (5 минут)

### Уровень 2. Quick Resto (Pro-пакет)
- **API:** REST, документация — https://docs.quickresto.ru/
- **Авторизация:** API-ключ
- **Эндпоинты:**
  - Получение списка блюд, категорий, модификаторов
- **Сложность:** низкая, аналог Poster
- **Что нужно от клиента:** API-ключ

### Уровень 3. iikoCloud (Enterprise-пакет)
- **API:** REST, документация — https://api-ru.iiko.services/
- **Авторизация:** iikoCloud API Login → получение токена (живёт 1 час)
- **Эндпоинты:**
  - `POST /api/1/nomenclature` — полная номенклатура (блюда, группы, модификаторы)
  - `POST /api/1/stop_lists` — стоп-лист (что закончилось)
- **Сложность:** средняя (токен живёт час, нужно обновлять; XML-like структура ответа)
- **Что нужно от клиента:** лицензия iikoCloud (обычно уже включена), API-логин
- **Нюанс:** у многих клиентов iiko Server (локальная), а не Cloud — тогда нужен iikoTransport, это сложнее

### Уровень 4. r_keeper 7 (не делаем в v1)
- **API:** XML Interface, работает через локальный сервер
- **Авторизация:** нужен модуль «Интеграция» на сервере клиента
- **Сложность:** высокая — нужен доступ к серверу, установка модуля обычно через интегратора UCS
- **Решение:** если клиент на r_keeper — предлагаем Google Sheets

### Уровень 5. 1С:Ресторан / 1С:УНФ (не делаем в v1)
- **API:** OData через HTTP-сервис 1С
- **Сложность:** высокая — у каждого своя конфигурация
- **Решение:** Google Sheets

## Архитектура синхронизации

```
[Cloudflare Cron Trigger: раз в сутки, 04:00]
          ↓
[Cloudflare Worker: sync-menu]
     ├→ читает POS API (или Google Sheets)
     ├→ нормализует в единый формат JSON
     ├→ записывает в Cloudflare KV (кеш)
     └→ логирует: «синхронизировано N позиций»
          ↓
[Cloudflare KV: menu-cache]
     ↓
[PWA читает из KV через Worker API]
```

**Единый формат меню (JSON):**
```json
{
  "restaurant": "Название",
  "updated": "2026-04-10T04:00:00Z",
  "categories": [
    {
      "name": "Горячее",
      "items": [
        {
          "name": "Стейк рибай",
          "description": "Мраморная говядина, 300г, подаётся с овощами гриль",
          "price": 1890,
          "photo": "https://...",
          "allergens": ["глютен"],
          "tags": ["мясо", "гриль"],
          "available": true
        }
      ]
    }
  ]
}
```

**Если POS API не отвечает** → Worker берёт предыдущий кеш из KV. Меню продолжает работать с вчерашними данными. Ничего не ломается.
